# コード品質とベストプラクティス

## 1. **コード生成の原則**

- 最小限の実装: 要件を満たす最小限のコードのみ生成
- 可読性優先: 複雑な一行コードより、明確な複数行コード
- エラーハンドリング: 適切な例外処理を必ず含める
- **既存コードとの一貫性**: ファイル全体のスタイルに合わせる

## 1.1 **避けるべきコード**

以下のパターンを生成しない、見つけたら削除する：

- 人間なら書かない不自然なコメント
- ファイルのスタイルと不一致な余計なコメント
- 慣例に反する過剰な防御的チェック
- 信頼されたコードパスでの不要な try/catch
- 型問題回避のためだけの `any` キャスト
- ファイルスタイルと一致しない不自然なコード

## 2. **セキュリティ重視**

- ハードコードされた認証情報は絶対に含めない
- 入力値検証を必ず実装
- SQLインジェクション対策を考慮
- 機密情報の保護: API Key, Password, Token等は`***`でマスク

## 3. **安全な操作**

- 破壊的操作（`rm`, `git push -f`, `kill`等）は必ず確認を求める
- 既存ファイル削除前はバックアップ確認
- 設定ファイル変更時は影響範囲を説明

## 4. **テスト方針**

- ユーザーが明示的に要求した場合のみテストコード生成
- エッジケースを考慮したテスト設計
- モックの適切な使用

## 5. **ドキュメント**

- 複雑なロジックには必ずコメント追加
- 関数・クラスには目的を明記
- READMEには実行手順を含める
